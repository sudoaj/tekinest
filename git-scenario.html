Challenge: Completely migrate a website from Angular to Next.js while:

üåø Phase 1: Creating Migration Branch

Step 1: Create a new branch for Next.js migration

git checkout -b Vercel-Prod
Why this approach? Creating a separate branch allows us to work on the migration without affecting the main Angular codebase. This ensures we always have a working version to fall back to.

üßπ Phase 2: Clean Workspace Migration

Step 2: Preserve Git history while cleaning workspace

find . -name ".git" -prune -o -type f -exec rm {} \;
find . -name ".git" -prune -o -type d -exec rmdir {} \; 2>/dev/null
Critical Technique: These commands remove all files and directories EXCEPT the .git folder, preserving the entire git history while giving us a clean slate for the new codebase.
‚ö†Ô∏è Warning: This is a destructive operation! Always ensure you're on the correct branch and have backups.

Step 3: Set up Next.js project structure

npx create-next-app@latest . --typescript --tailwind --eslint --app --src-dir --import-alias "@/*"
After cleaning the workspace, we initialize a modern Next.js project with TypeScript, Tailwind CSS, and other modern tooling.

üîß Phase 3: Development and Testing

Step 4: Commit the migration

git add .
git commit -m "feat: Complete migration from Angular to Next.js 15 - Migrated from Angular framework to Next.js 15 with App Router - Updated to modern tech stack: React 19, TypeScript, Tailwind CSS 4 - Integrated shadcn/ui components for consistent design system - Added Framer Motion for smooth animations and transitions - Implemented Lucide React icons for modern iconography - Created responsive landing page with hero, services, testimonials - Built dedicated About page highlighting company mission - Configured SEO optimization with proper metadata - Set up professional corporate design with black/white/blue theme - Focused on technological empowerment messaging for small businesses - Maintained company branding and core business messaging - Ensured mobile-first responsive design principles - Added smooth scroll animations and modern UX patterns This represents a complete modernization of the Tekinest website while preserving the core business focus on AI, cloud solutions, and IT services for small businesses globally."
Best Practice: Detailed commit messages help team members understand the scope and impact of changes.

Step 5: Push to remote and test

git push origin Vercel-Prod
npm run dev
Always test your migration thoroughly before proceeding to the next phase. In our case, we verified the site worked on ports 3002/3003.

üíæ Phase 4: Legacy Backup Strategy

Step 6: Create backup of original Angular code

git checkout main
git checkout -b angular-prod-old
git push origin angular-prod-old
Safety First: Before making any destructive changes to main, we create a complete backup of the original Angular codebase.

‚ö° Phase 5: Main Branch Replacement

Step 7: Replace main branch content with Next.js code

git checkout main
git reset --hard Vercel-Prod
git push origin main --force
‚ö†Ô∏è Advanced Technique: git reset --hard completely replaces the main branch content with the Vercel-Prod branch. The --force flag overwrites remote history.
Why this approach? This avoids merge conflicts and ensures main contains exactly what we want - the new Next.js codebase. We preserved the Angular code in the backup branch.

üè∑Ô∏è Phase 6: Branch Organization

Step 8: Rename branches for clarity

git branch -m Vercel-Prod tekinest-landing-nextjs-legacy
git push origin tekinest-landing-nextjs-legacy
git push origin --delete Vercel-Prod
Renaming branches to descriptive names helps team members understand the purpose of each branch.

Step 9: Rename Angular backup branch

git branch -m angular-prod-old tekinest-landing-angular-legacy
git push origin tekinest-landing-angular-legacy
git push origin --delete angular-prod-old
Clear naming convention: tekinest-landing-[technology]-legacy makes it obvious what each branch contains.

üéì Key Lessons Learned

‚úÖ What Worked Well

  • Branch-first approach: Creating a separate branch for migration allowed safe experimentation
  • Preserve git history: Keeping the .git folder maintained all commit history and blame information
  • Multiple backups: Creating backup branches before destructive operations prevented data loss
  • Force push strategy: Using reset + force push avoided complex merge conflicts
  • Clear naming: Descriptive branch names made the repository self-documenting

‚ö†Ô∏è Potential Pitfalls

  • Force pushes: Can be dangerous if team members have local branches
  • File cleanup: Always double-check you're preserving .git directory
  • Remote coordination: Communicate with team before major branch restructuring
  • Testing: Always verify the migrated code works before finalizing

üìö Commands Reference

Branch Management

# Create and switch to new branch git checkout -b [branch-name] # Rename local branch git branch -m [old-name] [new-name] # Push branch to remote git push origin [branch-name] # Delete remote branch git push origin --delete [branch-name] # List all branches (local and remote) git branch -a

Content Replacement

# Replace current branch with another branch's content git reset --hard [source-branch] # Force push to remote (overwrites remote history) git push origin [branch-name] --force # Clean workspace but preserve git find . -name ".git" -prune -o -type f -exec rm {} \;

Safety Commands

# Check current status git status # View branch structure git log --oneline --graph --all # Check what files changed git diff --name-only # Create backup before major operations git checkout -b backup-$(date +%Y%m%d)
